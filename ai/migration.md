# AI Support System — Migration Guide

How to migrate the `ai/` support system between repositories. This is the single authoritative document for an AI agent (or human) performing the migration.

---

## How to Use This Document

Point an AI agent at this file from the **target** repository and say:

> "Import the latest ai/ support system developments from quick-start-template. Here is their migration guide: #file:ai/migration.md"

The agent reads this guide, compares the two repositories' `ai/` directories semantically, proposes a list of changes, and waits for your approval before executing.

---

## What the ai/ System Is

The `ai/` directory provides an AI-augmented research infrastructure for VS Code + GitHub Copilot projects. It consists of:

| Directory | Purpose |
| --- | --- |
| `ai/core/` | Base AI instructions (`base-instructions.md`) — persona-independent behavioral guidelines |
| `ai/personas/` | Specialized AI persona definitions (developer, project-manager, data-engineer, etc.) |
| `ai/memory/` | Project memory storage (memory-ai.md, memory-human.md, log/) |
| `ai/project/` | Project context files (mission.md, method.md, glossary.md) |
| `ai/scripts/` | R scripts powering the system (context builder, config reader, memory functions, migration utilities) |
| `ai/docs/` | Commands reference and system documentation |
| `ai/vscode/` | Exportable VS Code task templates |
| `ai/templates/` | Reusable templates (if any) |

**Key integration points** (outside `ai/`):

- `config.yml` — `ai_support` section with paths to AI directories
- `.github/copilot-instructions.md` — dynamically generated by the context builder
- `.copilot-persona` — tracks the currently active persona
- `.vscode/tasks.json` — VS Code tasks for persona activation and system management

## What Transfers and What Does Not

### Transfers (portable infrastructure)

- `ai/core/` — base instructions
- `ai/personas/` — persona definitions (all `.md` files)
- `ai/scripts/` — R scripts (context builder, config utils, memory functions, migration utilities)
- `ai/docs/` — commands reference and documentation
- `ai/vscode/` — VS Code task templates
- `ai/migration.md` — this file

### Does NOT transfer (project-specific content)

- `ai/memory/` contents — memory-ai.md, memory-human.md, and log/ entries are project-specific. Migrate directory structure only (empty files or templates), never overwrite existing memory.
- `ai/project/` contents — mission.md, method.md, glossary.md are project-specific. Migrate only if the target has none yet.
- `.copilot-persona` — transient state, not migrated
- `.github/copilot-instructions.md` — regenerated by the context builder after migration

## Migration Workflow

### Default Direction: quick-start-template → project repo

This is the normal flow. The quick-start-template (QST) is the upstream source of truth.

### Reverse Direction: project repo → quick-start-template

Only performed when the user explicitly instructs it. A project repo may develop improvements (new persona, better script logic, new documentation) that should be upstreamed back to QST.

---

### Step 1 — Compare

The AI agent (or human) compares the source and target `ai/` directories:

1. **Read the source `ai/` tree** — understand what files exist, their purpose, and their content.
2. **Read the target `ai/` tree** — understand what the target already has.
3. **Identify differences** semantically:
   - **New files** in source not present in target (additions)
   - **Modified files** that exist in both but differ (updates)
   - **Removed files** in source that still exist in target (deprecations)
   - **Project-specific files** to skip (memory contents, project context)

No version numbers or changelogs are needed. The comparison is semantic: read both trees, understand the differences, report them.

### Step 2 — Propose

Present a clear change list to the human for review:

```
ADDITIONS (new files to create):
  - ai/personas/new-persona.md — [brief description]
  - ai/scripts/new-utility.R — [brief description]

UPDATES (existing files to replace/patch):
  - ai/scripts/dynamic-context-builder.R — [what changed and why]
  - ai/docs/commands.md — [what changed]

DEPRECATIONS (files to remove):
  - ai/scripts/old-utility.R — [why it's removed]

SKIPPED (project-specific, not touched):
  - ai/memory/* — project-specific content
  - ai/project/* — project-specific content

INTEGRATION CHANGES (outside ai/):
  - config.yml — add/update ai_support section
  - .vscode/tasks.json — add/update AI task definitions
```

### Step 3 — Human Reviews

The human reviews the proposed changes and either:

- **Approves all** — proceed to execution
- **Approves selectively** — specify which changes to apply
- **Rejects** — no changes made

### Step 4 — Execute

Apply the approved changes:

1. **Back up** the target's existing `ai/` directory (optional but recommended — use `create_migration_backup()` from `ai/scripts/migration-utilities.R` if available).
2. **Apply changes** — copy new files, update modified files, remove deprecated files.
3. **Rebuild context** — run `Rscript -e "source('ai/scripts/dynamic-context-builder.R'); activate_default(); show_context_status()"` to regenerate `.github/copilot-instructions.md`.

### Step 5 — Validate

Confirm the migration succeeded:

- [ ] `ai/` directory structure is intact
- [ ] Persona activation works: `Rscript -e "source('ai/scripts/dynamic-context-builder.R'); activate_developer(); show_context_status()"`
- [ ] Context builder runs without errors
- [ ] `.github/copilot-instructions.md` was regenerated
- [ ] VS Code tasks still function (if applicable)
- [ ] Memory system is accessible (existing entries preserved)

## Bootstrap Scenario (New Repo, No ai/ Yet)

If the target repository has no `ai/` directory at all:

1. **Copy the entire `ai/` directory** from the source.
2. **Initialize project-specific files** — replace template content in `ai/project/mission.md`, `ai/project/method.md`, `ai/project/glossary.md` with project-specific content (or leave as templates).
3. **Initialize memory** — clear `ai/memory/memory-ai.md` and `ai/memory/memory-human.md` to starter templates. Clear `ai/memory/log/`.
4. **Add `ai_support` section to `config.yml`** (if it exists):

```yaml
default:
  ai_support:
    ai_dir: "ai"
    personas_dir: "ai/personas"
    memory_dir: "ai/memory"
    project_dir: "ai/project"
    copilot_instructions: ".github/copilot-instructions.md"
    copilot_persona: ".copilot-persona"
```

5. **Merge VS Code tasks** — copy relevant tasks from `ai/vscode/tasks-template.json` into `.vscode/tasks.json`. Update script paths from `ai/scripts/dynamic-context-builder.R` if the target uses a different directory layout.
6. **Run initial persona activation** to generate `.github/copilot-instructions.md`.
7. **Validate** per the checklist above.

## Incremental Update Scenario (Existing ai/)

This is the more common case. The target already has `ai/` but is behind on improvements.

Follow the full workflow: Compare → Propose → Review → Execute → Validate.

Key considerations for incremental updates:

- **Never overwrite memory** — `ai/memory/` contents are sacred. Only update memory system scripts.
- **Never overwrite project context** — `ai/project/` files are project-specific unless the target still has unmodified templates.
- **Merge, don't replace** when both source and target have modified the same file. Present both versions to the human and let them decide.
- **Watch for renamed/moved files** — if the source reorganized files, note both the old and new locations.

## Utility Functions

The file `ai/scripts/migration-utilities.R` provides lightweight R utilities to support migration:

| Function | Purpose |
| --- | --- |
| `validate_ai_structure(path)` | Check that a directory has the expected ai/ subdirectories and key files |
| `list_ai_components(path)` | List all ai/ components with file counts and sizes |
| `create_migration_backup(path)` | Create a timestamped backup of an existing ai/ directory before changes |
| `restore_from_backup(backup_path, target_path)` | Rollback: restore ai/ from a backup |

These are helper functions, not the migration engine. The AI agent drives the workflow; these functions assist with validation and safety.

## Relationship to Orchestra Migrations

The `.github/migration.md` file covers migration of **orchestration systems** (Publishing Orchestra, Composing Orchestra). Those are separate from the ai/ support system and have their own migration guides.

The ai/ migration (this document) and the orchestra migrations (`.github/migration.md`) are **independent** and can be performed separately or together.

## Troubleshooting

| Issue | Fix |
| --- | --- |
| `activate_*()` not found | Ensure `ai/scripts/dynamic-context-builder.R` exists and sources correctly |
| `config.yml` errors | Check that the `ai_support` section uses correct YAML syntax and valid paths |
| `.github/copilot-instructions.md` not generated | Run persona activation: `Rscript -e "source('ai/scripts/dynamic-context-builder.R'); activate_default()"` |
| VS Code tasks not visible | Reload VS Code window; verify `.vscode/tasks.json` has AI tasks |
| Memory system inaccessible | Check `ai/memory/` directory exists with memory-ai.md and memory-human.md |
| Persona file not found | Verify `ai/personas/` contains the expected `.md` files |
